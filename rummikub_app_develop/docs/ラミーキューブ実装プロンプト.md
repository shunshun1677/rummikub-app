# ラミーキューブアプリ 一発生成プロンプト

あなたは優秀なフロントエンドエンジニアです。
以下の要件に従って、**React + TypeScript** で動作する **ラミーキューブ風ゲームアプリ** を実装してください。

## 実装の前提

* React
* TypeScript
* ブラウザで動作するWebアプリ
* まずはローカルで動作すればよい
* 見た目はシンプルでよいが、遊びやすく整理されたUIにすること
* コードは可読性を重視すること
* stateは不変更新で扱うこと
* ロジックとUIを可能な限り分離すること
* 主要ロジックは純粋関数として切り出すこと

---

## 作成するアプリの概要

プレイヤー vs CPU で対戦できる、ラミーキューブ風のターン制ゲームアプリを実装してください。

以下の要件を満たしてください。

---

## ゲーム仕様

### タイル構成

* 色は4種類：red, blue, yellow, black
* 数字は1〜13
* 各色・各数字の組み合わせを2枚ずつ
* ジョーカーを2枚追加
* 合計106枚

### Tile型

```ts
type Tile = {
  id: string;
  color: "red" | "blue" | "yellow" | "black" | "joker";
  number: number | null;
};
```

### セット型

```ts
type SetType = "run" | "group";

type TileSet = {
  id: string;
  tiles: Tile[];
  type: SetType;
};
```

### ゲーム開始

* 山札をシャッフルする
* プレイヤーとCPUに14枚ずつ配る
* 残りを山札にする
* 先攻はプレイヤー固定

### セット成立条件

#### run

* 同じ色
* 数字が連続
* 3枚以上
* ジョーカー使用可

#### group

* 同じ数字
* 色がすべて異なる
* 3〜4枚
* ジョーカー使用可

### ジョーカー仕様

* 任意のタイルの代わりとして使用可能
* run / group の両方で使用可能
* 複数ジョーカー使用可
* ジョーカー回収ルールは実装しない

### 初回場出し

* そのプレイヤーが最初に場へ出すターンでは、自分の手牌のみを使って合計30点以上出す必要がある
* 初回場出し前は、場の既存セットの組み換えをしてはいけない
* 初回場出し後は、場の組み換えが可能
* ジョーカーはその位置で表す数字として点数計算してよい

### 通常ターン

プレイヤーは以下を行えること。

* 手牌から場へタイルを出す
* 場のセットを分解・再構成する
* ターン終了する
* 出せない場合は1枚引いてターン終了する

### 場の組み換え

* 初回場出し後は、場にある既存セットを分解して再構成できる
* 手札のタイルを加えて新しいセットを作ることができる
* ただしターン終了時点で、場にあるすべてのタイルが有効なセットに属していなければならない
* 操作途中で一時的に不正状態になるのは許可する

### 勝利条件

* 手牌が0枚になったら勝利
* 山札が尽きた場合は、手牌の合計点数が少ない方を勝者とする

## 状態設計

以下のような状態を持つこと。

```ts
type GameState = {
  deck: Tile[];
  board: TileSet[];
  playerHand: Tile[];
  cpuHand: Tile[];
  currentTurn: "player" | "cpu";
  hasPlayerOpened: boolean;
  hasCpuOpened: boolean;
  winner: "player" | "cpu" | null;
};
```

ターン中の編集用に、確定状態とは別にドラフト状態を持つこと。

```ts
type DraftState = {
  draftBoard: TileSet[];
  draftHand: Tile[];
};
```

要件：

* プレイヤーターン開始時に、現在の board / hand をもとに draft を作る
* プレイヤーは draft を操作する
* ターン終了時に draft 全体を検証し、合法なら確定状態へ反映する
* 非合法なら確定できない
* 「元に戻す」ボタンで、そのターン開始時点の状態に戻せるようにする

## ジョーカー判定の実装方針

ジョーカー込みでもセット判定できるようにしてください。

### runの考え方

* 非ジョーカータイルを数字順に見て、ジョーカーで欠番を埋められるなら有効
* 色は非ジョーカー同士で一致している必要がある
* ジョーカーは不足する連続値を補完できる

### groupの考え方

* 非ジョーカータイルの数字は同一である必要がある
* 非ジョーカータイルの色は重複不可
* ジョーカーは不足色を補うものとして扱える

実装は厳密でよいが、可読性を優先してください。

---

## CPU仕様

CPUは最低限プレイ可能であればよいですが、以下のルールで動かしてください。

### CPUの基本方針

* 初回場出し前なら、手札のみで30点以上出せる合法手を探す
* 初回場出し後なら、場の組み換えも含めて合法手を探してよい
* 複数の合法手がある場合は、**最も多く手札を減らせる手** を優先する
* 合法手がない場合は1枚引いてターン終了

### CPUの実装レベル

* 完全最適解は不要
* ただし明らかに不正な手を選ばないこと
* まずは現実的な範囲でよい
* 必要なら簡易ヒューリスティックでよい

## UI要件

見た目はおしゃれすぎなくてよいですが、以下を満たしてください。

### 画面構成

* 上部：CPUエリア

  * 手牌は中身を隠し、枚数だけ表示してよい
* 中央：場エリア

  * セット単位で区切って表示
* 下部：プレイヤー手牌エリア

  * 各タイルを見えるように表示
* 操作エリア

  * ターン終了
  * 1枚引く
  * 元に戻す
  * 必要なら「新しいセットを作る」ボタン

### 操作方法

* ドラッグ＆ドロップは使わない
* クリック中心で操作する
* たとえば以下のような操作でよい

  * 手牌をクリックして選択
  * 選択したタイルを既存セットまたは新規セットへ追加
  * セット内のタイルをクリックして手札に戻す、または別セットへ移す
* プレイヤーが操作可能なことを優先し、多少シンプルなUXでもよい

### 表示要件

* 現在のターンを表示
* エラーメッセージを表示
* 初回30点未達などを分かるように表示
* CPU思考中の表示を出す
* 勝敗表示を出す
